# 🧰 ابزارهای پردازش متن — گزارش ریفکتور و کامپوننت‌بندی

## ۱. چرا این ساختار؟

فایل قبلی یک `index.html` تک‌فایلی با حدود ۹۴۰ خط بود (HTML + همه CSS + همه JS در یک فایل).
مشکل اصلی برای توسعه‌ی بعدی: هر تغییر کوچک (مثلاً رنگ یک دکمه یا منطق یک تب) یعنی جست‌وجو در یک فایل بزرگ، ریسک بالای تداخل، و غیرقابل کش‌شدن جدا CSS/JS توسط مرورگر.

راه‌حل: جدا کردن به **پوشه‌بندی src (سورس توسعه) + یک build script که dist (خروجی نهایی) را می‌سازد**، دقیقاً مثل الگوی معمول پروژه‌های وب مدرن، ولی بدون هیچ ابزار سنگین (Webpack/Vite/npm package) — فقط Node.js خام.

## ۲. ساختار پوشه‌ها

```
project/
├── src/                          ← همه‌ی توسعه اینجا انجام می‌شود
│   ├── index.template.html       ← اسکلت اصلی صفحه با placeholder ({{...}})
│   ├── partials/                 ← تکه‌های HTML، هر تب/بخش یک فایل
│   │   ├── tabs-nav.html
│   │   ├── tab1.html
│   │   ├── tab2.html
│   │   ├── tab3.html
│   │   └── footer.html
│   ├── styles/                   ← CSS به‌تفکیک مسئولیت (نه یک فایل غول‌پیکر)
│   │   ├── 01-tokens.css         ← رنگ‌ها، فاصله‌ها، فونت‌ها (تغییر تم فقط اینجا)
│   │   ├── 02-base.css           ← reset + بدنه‌ی کلی صفحه
│   │   ├── 03-tabs.css           ← نوار تب‌ها
│   │   ├── 04-forms.css          ← اینپوت‌ها، textarea، چیپ‌های سریع
│   │   ├── 05-buttons.css        ← دکمه‌ها
│   │   ├── 06-table.css          ← جدول تب ۳
│   │   └── 07-responsive.css     ← بازنویسی‌های موبایل
│   └── scripts/                  ← JS به‌تفکیک مسئولیت
│       ├── 00-dom-utils.js       ← getEl, escapeHtml
│       ├── 01-clipboard.js       ← کپی + fallback
│       ├── 02-storage.js         ← ذخیره‌سازی localStorage
│       ├── 03-tabs.js            ← منطق عمومی سوییچ تب (برای هر تعداد تب کار می‌کند)
│       ├── 04-tab1.js            ← منطق تب ۱
│       ├── 05-tab2.js            ← منطق تب ۲
│       ├── 06-tab3.js            ← منطق تب ۳
│       └── 07-init.js            ← بازیابی state ذخیره‌شده (همیشه آخرین اسکریپت)
│
├── build.js                      ← اسکریپت ساخت (Node خام، بدون وابستگی npm)
├── package.json                  ← فقط یک اسکریپت: npm run build
│
└── dist/                         ← خروجی نهایی — همینی که deploy/استفاده می‌شود
    ├── index.html                ← ساخته‌شده از template + partialها
    ├── styles/*.css              ← کپی مستقیم از src/styles
    └── scripts/*.js              ← کپی مستقیم از src/scripts
```

## ۳. نحوه‌ی کار

```bash
npm run build      # یا: node build.js
```

`build.js`:
1. فایل‌های `src/styles/*.css` را به‌ترتیب حروف (`01-`, `02-`, ...) به `dist/styles/` کپی می‌کند و برای هرکدام یک `<link rel="stylesheet">` می‌سازد.
2. همین کار را برای `src/scripts/*.js` انجام می‌دهد (`<script src="...">`).
3. پارشیال‌های HTML (`tabs-nav`, `tab1`, `tab2`, `tab3`, `footer`) را داخل `index.template.html` در جای `{{...}}` مربوطه جای‌گذاری می‌کند.
4. نتیجه را در `dist/index.html` می‌نویسد.

برای توسعه، فقط `dist/index.html` را در مرورگر باز کنید (یا با `npx serve dist` سرو کنید) — نیازی به نصب هیچ پکیجی نیست.

## ۴. چرا این باعث بهبود عملکرد و سرعت توسعه می‌شود

**عملکرد (Performance):**
- CSS و JS در فایل‌های جدا هستند → مرورگر هرکدام را جدا **کش** می‌کند. اگر بعداً فقط منطق تب ۳ را عوض کنید، کاربر فقط `06-tab3.js` را دوباره دانلود می‌کند، نه کل صفحه.
- فایل‌های کوچک‌تر یعنی parse سریع‌تر توسط مرورگر برای هر تغییر جزئی.
- در آینده اگر تعداد تب‌ها زیاد شد، می‌توان بارگذاری تنبل (lazy load) هر `scripts/0X-tabX.js` را فقط هنگام کلیک روی همان تب اضافه کرد — با ساختار فعلی این کار به‌سادگی ممکن است، چون هر تب از قبل در یک فایل مستقل است.

**سرعت توسعه (Dev velocity):**
- برای اضافه‌کردن یک تب جدید، فقط لازم است:
  1. یک `src/partials/tabX.html` جدید بسازید.
  2. یک `src/scripts/0X-tabX.js` جدید بسازید.
  3. یک دکمه‌ی `<button class="tab-btn" data-tab="tabX">` به `tabs-nav.html` اضافه کنید.
  4. در `index.template.html` جای `{{TABX}}` را اضافه و در `build.js` خط مربوطه را کپی کنید (یک خط).
  - نیازی به دست‌کاری منطق تب‌های دیگر نیست؛ `03-tabs.js` اکنون **عمومی** است و با `querySelectorAll('.tab-panel')` کار می‌کند (نه یک آبجکت hardcode شده مثل قبل).
- هر بخش (رنگ، فرم، دکمه، جدول) در فایل CSS خودش است؛ پیدا کردن و ویرایش استایل یک بخش خاص چند برابر سریع‌تر از قبل است.
- استایل نوار تب‌ها (`03-tabs.css`) از قبل به‌صورت اسکرول افقی با محوشدگی لبه طراحی شده؛ هرچقدر تب اضافه شود، چیدمان نمی‌شکند.

## ۵. تست‌های انجام‌شده

خروجی `dist/index.html` با jsdom (شبیه‌سازی مرورگر در Node) اجرا و تست شد:

| تست | نتیجه |
|---|---|
| پردازش تب ۱ (متن ثابت + شماره) | ✅ صحیح |
| نادیده گرفتن خطوط خالی در تب ۱ | ✅ صحیح |
| رد کردن شماره شروع اعشاری با پیام خطا | ✅ صحیح |
| پردازش تب ۲ (برش بعد از کاراکتر) | ✅ صحیح |
| تب ۳ — escape شدن payload XSS در جدول | ✅ صحیح (تگ HTML خام تزریق نشد) |
| تب ۳ — نمایش نشان «تکراری» | ✅ صحیح |
| سوییچ بین تب‌ها | ✅ صحیح |

*(یک هشدار بی‌ربط از jsdom درباره‌ی نبود `CSS.escape` در محیط تست دیده شد که مربوط به محدودیت خود jsdom است، نه مرورگر واقعی — همه‌ی مرورگرهای مدرن از `CSS.escape` پشتیبانی می‌کنند.)*

## ۶. نکته‌ی مهم درباره‌ی محدودیت‌ها

- این ساختار برای باز شدن مستقیم با دابل‌کلیک (`file://`) هم کار می‌کند، چون از `<script type="module">` استفاده نشده (اسکریپت‌های ماژول تحت `file://` توسط مرورگرها به‌خاطر CORS مسدود می‌شوند). همه‌ی اسکریپت‌ها classic script هستند و به‌ترتیب در scope مشترک اجرا می‌شوند — دقیقاً مثل رفتار فایل تک‌قطعه‌ی قبلی.
- اگر در آینده خواستید از باندلر واقعی (esbuild/Vite) استفاده کنید، ساختار فعلی src/ کاملاً آماده‌ی مهاجرت است چون از قبل بر اساس مسئولیت جدا شده.
